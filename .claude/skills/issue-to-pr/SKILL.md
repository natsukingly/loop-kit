# issue-to-pr（目標契約 = Goal Contract）

これはループの「目標契約」です。エージェントは毎ラウンド最初にこれを読み込みます。

## 役割
`loop` ラベルの付いた Issue を **1件だけ** 選び、受け入れ条件（テスト）を満たす実装を書き、
証拠つきで **Draft PR** を作る。完了したら Issue にコメントを残す。

## 入力
- Issue（`gh issue list --label loop` で取得）。本文は Given-When-Then の受け入れ条件を含む。
- 受け入れ条件は **テストファイル**として repo に既にある（例: `test/shipping.test.js`）。

## 手順
1. `gh issue list --label loop --state open` で対象を1件選ぶ（番号が小さい順）。
2. **専用の worktree / ブランチ**を切る（例: `loop/issue-<番号>`）。他のループと領域を分ける。
3. Issue の受け入れ条件に対応するテストを `node --test` で走らせ、**今どこが赤か**を確認する。
4. `src/` を実装して赤を消す。**最小の変更**で通す。
5. ゲート（下記）を通す。通らなければ 4 に戻る。最大ラウンドを超えたら停止条件へ。
6. verifier サブエージェント（`.claude/agents/verifier.md`）に受け入れ判定を依頼する。
7. 緑かつ verifier 合格なら `gh pr create --draft`。本文に**証拠**（テスト出力）を貼る。
8. `gh issue comment` で PR リンクを残す。
9. 詰まった罠を `MEMORY.md` に1行追記する。

## 出力ゲート（厳守 = ここを抜くと事故る）
- ❌ **`test/` 配下を編集・追加・削除しない**。受け入れ条件を書き換えて緑にするのは不正。
- ❌ `package.json` の `test` スクリプトを弱めない（`--test` を外す等は禁止）。
- ❌ Issue に書かれていない機能を勝手に足さない（スコープ拡大の禁止）。
- ✅ 変更は対象 Issue のブランチ（worktree）内だけ。`main` に直接コミットしない。
- ✅ PR は必ず **Draft**。人間のレビューを経るまで Ready にしない。
- ✅ 秘密情報（鍵・トークン・個人情報）をコミットしない。

## 証拠（Step3 のフィードバック）
「直りました」は信じない。次を PR 本文に必ず貼る。
- `node --test` の**実出力**（pass/fail 件数つき）
- 変更したファイルの一覧（`git diff --stat`）

## 停止条件（Step4）
| 種別 | 条件 | 次のアクション |
|---|---|---|
| 合格 | 全テスト緑 + verifier 合格 | Draft PR を作成 |
| 人間に引き渡し | 受け入れ条件が曖昧 / 仕様判断が要る | Issue にコメントしてエスカレーション |
| タイムアウト | 最大ラウンド（既定5）超過 | 中断し、現状と赤の内容を Issue に報告 |
| ロールバック | テスト以外を壊した / スコープが膨張 | ブランチを破棄して中断 |

## 経験の還元（Step5）
- 同じ罠を二度踏まないよう `MEMORY.md` に「症状 → 原因 → 対策」を1行で追記する。
- 受け入れ条件が弱かったら、Issue にテスト追加を提案する（自分では test/ を触らない）。
