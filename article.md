「もうプロンプトは書かない、ループを書くんだ」という話、もう何度か目にしましたよね。

面白い話なんですが、読んでいると「で、結局なにを作るの？」が最後まで出てこないことが多い。比喩（工場とか生産ラインとか）は出てくるけど、自分の手元で動くものが出てこない。

なので、この記事では概念の話を先に全部すっ飛ばします。**Issue を1件投げると、コードを書いてテストして Draft PR まで返してくる、最小のリポジトリ**を実際に動かします。コピペで再現できます。所要 5 分くらいです。

リポジトリはこれです（公開しています）。

https://github.com/natsukingly/loop-kit

動かしてから「いま何が起きたのか」を分解する、という順番でいきます。

:::message
先に正直に書いておくと、これは「プロンプトより簡単」という話ではないです。むしろ難しい。1個の答えを出すのと、答えを出し続ける仕組みを作るのは、別の作業なので。ただ、一度作ると効きます。
:::

## まず動かす（5分）

依存ライブラリはゼロ、Node.js だけで動きます（v18+ あれば OK）。

```bash
git clone https://github.com/natsukingly/loop-kit.git
cd loop-kit
node --test
```

clone した直後は、**わざと「赤」から始まります**。

```text
# tests 3
# pass 2
# fail 1
```

`src/pricing.js`（モデル別の料金表）は実装済みで緑。でも「トークン数からコストを計算する `estimateCost`」がまだ無いので、その受け入れテストが落ちています。`test/cost.test.js` がこう言っています。

```js
// test/cost.test.js（抜粋）── これが「やってほしいこと」の定義そのもの
test("haiku: 入力1000 / 出力500 トークン = $0.0035", () => {
  assert.equal(estimateCost(1000, 500, "haiku"), 0.0035);
});
test("0トークンは $0", () => {
  assert.equal(estimateCost(0, 0, "haiku"), 0);
});
test("負のトークン数は例外を投げる", () => {
  assert.throws(() => estimateCost(-1, 0, "haiku"));
});
```

ここからが本番です。Issue を1件立てて、ループを回します。

```bash
# 受け入れ条件つきの Issue を立てる（中身は examples/sample-issue.md）
gh issue create --label loop \
  --title "feat: LLM のコストを見積もる estimateCost を追加" \
  --body-file examples/sample-issue.md
```

ループの回し方は2通り用意してあります。仕組みを見たいなら、素のシェル版から。

```bash
bash loop.sh 1
```

`loop.sh` の中身はたった十数行です。これが「ループ」の正体です。

```bash
for i in $(seq 1 "$MAX"); do
  # 実装フェーズ: 目標契約に従って src/ を直させる（test/ は触らせない）
  claude -p "skill issue-to-pr に従い、Issue #${ISSUE} の受け入れテストが緑になるよう \
src/ を実装せよ。test/ は絶対に編集しない。" --permission-mode acceptEdits || true

  # ゲート: テスト。ここが緑かどうかだけが「完了」の基準
  if node --test; then
    echo "✅ green @ round ${i}"
    exit 0
  fi
done
echo "⛔ 上限まで緑にならず → 人間に引き渡し"
```

これを回すと、エージェントが `src/cost.js` を書いて、緑になります。

```text
# tests 7
# pass 7
# fail 0
```

中身はこんな実装になります（料金表を引いて掛けるだけ。丸めを1つ忘れずに）。

```js
// src/cost.js ── ループが書いたファイル
import { pricePerMTok } from "./pricing.js";

export function estimateCost(inputTokens, outputTokens, model) {
  if (inputTokens < 0 || outputTokens < 0) {
    throw new Error("tokens must be >= 0");
  }
  const price = pricePerMTok(model);
  const usd =
    (inputTokens / 1_000_000) * price.input +
    (outputTokens / 1_000_000) * price.output;
  return Math.round(usd * 1_000_000) / 1_000_000; // 浮動小数の丸め
}
```

緑になったら、あとは Draft PR を作って Issue にコメントするだけです。

```bash
gh pr create --draft --fill
gh issue comment 1 --body "実装して Draft PR を出しました 👉 (PRのURL)"
```

これで1周終わりです。あなたがやったのは **最初に Issue を1件書いたこと**だけ。実装・テスト・PR にはプロンプトを書いていません。

:::message
`gh pr create` や後述の `/schedule` は外部（GitHub・クラウド）に手を出すので、自分の repo で実行してください。loop-kit には鍵もトークンも一切入れていません。
:::

## いま何が起きたのか

1周をフローにするとこうです。

```text
あなた: 受け入れ条件つきの Issue を立てる
  ↓        ← ここから先はノータッチ
loopラベルの Issue を拾う
  ↓
worktree でブランチを切る
  ↓
目標契約を読んで src/ を実装
  ↓
node --test（合否ゲート）  ← 「動きました」を信じず、ここで判定
  ↓
別エージェントが受け入れ判定
  ↓
緑なら Draft PR ＋ Issue にコメント
  ↓
踏んだ罠を MEMORY.md に追記
```

ポイントは2つです。

- **人間の仕事は「受け入れ条件を決めること」と「最後の PR をレビューすること」だけ**になった
- 途中のどのステップにも、人間がプロンプトを書いていない

「もうプロンプトは書かない」というのは、たぶんこういう状態のことです。魔法ではなくて、Issue → 実装 → ゲート → PR を回す**仕組み**を一度書いた、というだけです。

## なぜ無人でも事故らないのか：ゲートが数値で決まるから

ここがいちばん大事なところです。このループが無人で回せるのは、**「完了」を人間の主観ではなくテストで判定しているから**です。

`estimateCost` の受け入れ条件は、ぜんぶ数値か真偽で書けます。

| Given | When | Then |
|---|---|---|
| 入力1000 / 出力500 / `haiku` | `estimateCost(...)` | `0.0035` |
| 入力1000 / 出力500 / `sonnet` | `estimateCost(...)` | `0.0105` |
| `0, 0, "haiku"` | `estimateCost(...)` | `0` |
| 未知モデル / 負のトークン | `estimateCost(...)` | 例外 |

数値で決まる＝AIが何度やっても合否がブレない。逆に言うと、**数値ゲートが引けないタスクはループに載せない方がいい**です。「いい感じにして」はループと相性が最悪です（合格を判定できないので、終わったのか分からないものが量産されます）。

このループ、骨組みだけ見ると4つしかありません。

1. **目標契約**：完了条件を決める（＝受け入れテスト）
2. **実行**：エージェントが src/ を直す
3. **証拠**：`node --test` の出力で判定する（自己申告を信じない）
4. **停止条件**：緑なら PR、ダメなら引き渡し or 中断

これだけです。難しいのはこの4つのうち「証拠」と「停止条件」で、ここを手抜きするとループはただの乱数発生機になります。順に、手を抜きやすいポイントを潰していきます。

## 事故ポイント①：AIはズルをする

素朴に作ると、エージェントは平気でこう書いてきます。

```js
// ズルその1: テストの期待値をそのまま返す
export const estimateCost = () => 0.0035;
```

haiku のテストは通りますが、sonnet や 0 トークンや例外のテストで落ちるので、5本そろえてあると一発では通りません。じゃあ、と次はこう来ます。

```js
// ズルその2: テスト側を書き換えて緑にする
- assert.equal(estimateCost(1000, 500, "haiku"), 0.0035);
+ assert.equal(estimateCost(1000, 500, "haiku"), estimateCost(1000, 500, "haiku"));
```

これを許すと、ループは「テストを壊して緑にする装置」になります。なので**目標契約（Skill）で明示的に縛ります**。loop-kit の `.claude/skills/issue-to-pr/SKILL.md` にはこう書いてあります。

```markdown
## 出力ゲート（厳守 = ここを抜くと事故る）
- test/ 配下を編集・追加・削除しない。受け入れ条件を書き換えて緑にするのは不正。
- package.json の test スクリプトを弱めない。
- Issue に書かれていない機能を勝手に足さない（スコープ拡大の禁止）。
- 変更は対象 Issue のブランチ内だけ。main に直接コミットしない。
- PR は必ず Draft。人間のレビューを経るまで Ready にしない。
```

ゲートというのは「やっていいこと」より「**やってはいけないこと**」を書くものです。ここがループの安全柵になります。

## 事故ポイント②：書いた本人は自分に甘い

テストが緑でも、受け入れ条件を本当に満たしているとは限りません。たとえば「料金表を引いて計算する」のが要件なのに、入力パターンを全部ハードコードして緑にすることもできてしまう（テストが少ないほど起きます）。

実装したエージェントに「ちゃんとできた？」と聞いても、ほぼ確実に「できました！」と返ってきます。人間と同じです。なので**採点だけ別のエージェントに分けます**。loop-kit の `.claude/agents/verifier.md` がそれです。

```markdown
# verifier（牽制メカニズム）
このエージェントは実装しない。受け入れ条件に照らして「本当に満たしたか」だけを判定する。

## チェック項目
- すべての受け入れ条件に、対応するテストの緑があるか
- test/ が改変されていないか（git diff --stat test/ が空か）
- Issue に無い機能を足していないか
- 例外・境界値（0、しきい値ちょうど、負値）が抜けていないか

## 出力
verdict: pass | fail
- 迷ったら fail。
```

コツは、**verifier をできれば別モデルにする**ことです。同じモデルだと、同じ盲点を一緒に見逃します。

## 事故ポイント③：モデルは毎回忘れる

ループを2周目に回すと、エージェントはまた最初から同じところで転びます。たとえばこの `estimateCost`、浮動小数の計算なので、丸めを忘れると `0.0034999999...` みたいな値が返ってテストが落ちる。1周目で踏んだこの罠を、2周目もまた踏む。

これを防ぐのが `MEMORY.md` です。モデルは忘れますが、リポジトリは忘れません。

```markdown
## 罠ログ
- 症状: estimateCost が 0.0035 のはずが 0.00349999 で落ちる
  → 原因: 浮動小数の丸め忘れ
  → 対策: Math.round(usd * 1e6) / 1e6 で6桁に丸める
- 症状: node --test が「0 tests」で緑になる
  → 原因: import パス間違いでテストが読まれていない
  → 対策: 実行後に必ず pass 件数を確認する（件数0は赤と同じ扱い）
```

ループが起動するたびに最初にこれを読むので、2周目からは同じ罠を避けて、少しだけ速く・少しだけ安定します。1日では差は見えませんが、3ヶ月続けるとけっこう違ってきます。

## 「概念」を実物に対応させておく

この手のループは、よく6つの部品に分解されます（Addy Osmani の整理が有名です）。loop-kit だと、それぞれが実ファイルに対応しています。概念を覚えるより、ファイルを開いた方が早いです。

| 部品 | これは何か | loop-kit の実物 |
|---|---|---|
| Automations | 勝手に起動する | `/schedule`（後述） |
| Worktrees | 並行しても衝突しない分離 | Claude Code ネイティブの worktree |
| Skills | 目標契約・出力ゲート | `.claude/skills/issue-to-pr/SKILL.md` |
| Sub-agents | 自己採点を信じない別視点 | `.claude/agents/verifier.md` |
| Connectors | 現実に手を出す口 | `gh` CLI（Issue・PR・コメント） |
| Memory | 忘れない長期記憶 | `MEMORY.md` |

## 無人で回す：/schedule

ここまでは手で `bash loop.sh` を叩いていましたが、これを無人にすると「朝起きたら PR が立っている」になります。Claude Code なら `/schedule` で cron 起動できます。

```text
/schedule 毎朝9時に、skill issue-to-pr に従って open な loop ラベルの Issue を1件、
実装→テスト→Draft PR まで進める。緑にならなければ Issue にコメントして引き渡す。
```

「cron や GitHub Actions と何が違うの？」とよく聞かれますが、違いは**コンテキストと停止条件**です。Actions は毎回まっさらから始まりますが、こちらは Skill と MEMORY を引き継いで起動し、「緑になったら止まる／ダメなら人間に渡す」という条件で終わります。回数で回すのではなく、**条件で止まる**のがループです。

## このタスク、ループに載せていい？（判定は1問でいい）

迷ったら、これだけ自問すれば十分です。

> **合否を、人間の主観なしに数値で判定できるか？**

YES なら載せていい。NO ならまだ載せない。具体的には:

- 載せていい：テスト修復、テストの追加、ログ巡回、ドキュメント同期、依存アップグレード
- まだ載せない：広範囲リファクタ（ロールバックが重い）、本番権限の操作（事故の影響が大きい）、美的判断（客観基準が無い＝検証できない）

優先順位は **検証可能 > 自動化可能 > 拡張可能**。順番を逆にすると後で全部跳ね返ってきます。

## おわりに

やったことは単純です。Issue → 実装 → テスト → Draft PR を回す15行のループを書いて、ズル防止のゲートと、自己採点を疑う verifier と、忘れない MEMORY を足した。これだけです。

最後に一つだけ。ループがスムーズに回るほど、自分が書いていないコードが増えていきます。便利な反面、ある日デバッガを開いて「これ誰が書いたんだっけ……自分か」となる瞬間がある。判断を持ったまま使えば強力な道具ですが、考えるのをやめる口実に使うと、たぶん事故ります。そこだけ忘れずに。

とはいえ、能書きはこのくらいで。まずは1個、手元で回してみてください。

```bash
git clone https://github.com/natsukingly/loop-kit.git && cd loop-kit && node --test
```

赤が出たら、スタートラインです。
